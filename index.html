Fourth meeting - Submitting a Patch

Introduction

This is the third activity in the Contribute to GNOME Apps series. Do you want to start from the beginning? Check out the first activity which will introduce you to choosing a GNOME App and getting involved in the GNOME community.

In this meeting participants learn will how to submit a patch that solves a bug for the App they chose in the first meeting.

Solve the Bug - 30'

During the last meeting, your group found a beginner-friendly bug in GNOME's bug tracker, and used tools and techniques to understand the code.

Now that you have identified the problem, you can start thinking about how to solve it.

When making changes there are a few guidelines

  1. Don't add any changes to the code unless they are related to the bug you are trying to fix. This includes style changes.
  2. Follow the coding style used in the code you are working on. If the project does not have an explicit "coding style" document, take a look at the surrounding code and try to use the same format.
  3. You can use git diff --check to make sure the patch does not contain trailing whitespace and spurious empty lines.

GTK+ also includes the following developer tools that may be helpful for you

  1. GTK+ Demo demonstrates different widgets and features of GTK+ and includes code examples of how they can be used. It can be opened from the terminal by typing
    • gtk3-demo

  2. The GTK+ Icon Browser is a list of icons shipped with GTK+. You can use it to discover both names of normal or symbolic icons. It can be opened from the terminal by typing
    • gtk3-icon-browser

  3. GTK+ Widget Factory is an example application demonstrating the majority of different widgets which can be created with GTK+. It can be opened from the terminal by typing
    • gtk3-widget-factory

Install Gitg 5'

GNOME uses Git as a tool to make and manage all the changes of the code.

Type the following command into your terminal to install Gitg on your laptop:

  • Debian-based linux systems: sudo apt-get install gitg
  • Red Hat-based linux systems: dnf install gitg

Follow up on feedback

Your patch will likely go through several iterations of patch review. Wait to get comments about how you can improve the patch, and then make modifications if they are necessary. Then create a new patch and attach it to the bug. This kind of review process is standard practice within GNOME and helps to ensure that our code is all high-quality. Once your patch looks good, the reviewer will merge it in.

If your patch is accepted in the first review, you don't need to do this step, at least for this patch. Wait until the maintainer pushes your commit to the master branch.

If you need to modify your patch we need to reset the code changes to the previous commit, but with your changes in the stage area. To do so we need a terminal command (copy your commit message since it will be removed):

  • git reset --soft HEAD^

Now change the files you need and put them in the stage area as we learnt and commit again. Now you can attach the patch to Bugzilla again and repeat the process until your patch is accepted.
